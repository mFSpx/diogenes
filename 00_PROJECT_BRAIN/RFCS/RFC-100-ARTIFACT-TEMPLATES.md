# RFC-100: Artifact Generation Templates

Status: DRAFT  
Subject ID: `artifact_templates`  
Normative role: defines artifact generation as deterministic, source-anchored, receipt-backed output construction; model prose may assist but must not own the artifact contract.


## 0. Claim Discipline / ABBA3^5 Gate

ABBA3^5 is a local operator instruction, not an established external domain term. In this RFC it means: do not let confident prose outrun receipts. Every material CLAIM in this RFC must survive these five checks before it is treated as system guidance:

1. **Claim-state:** classify the statement as CLAIM, observation, inference, hypothesis, suggestion, or confidence-rated candidate; factual CLAIMs need receipts or cited sources.
2. **Provenance-count-and-reason:** state which operator instruction, repo file/code, receipt, or external source backs the claim; if only a few docs were consulted, name the count and why those docs were chosen.
3. **Naming-integrity:** preserve names as integral to their statement/context; label local/metaphorical names as local instead of pretending they are established terms.
4. **Reuse-before-reinvention:** search the LUCIDOTA Dev Library and known established concepts before proposing new code; reinvent only for sovereignty, objective superiority, necessity, or explicit operator intent.
5. **Operational-proportionality:** security, logging, tests, audits, and refusal gates are slop if they freeze work, flood storage, or waste operator time without proportional risk/truth benefit.

## 1. Thesis

LUCIDOTA artifacts — reports, case packets, bundles, letters, briefs, exports, review surfaces — must be generated from bounded context and evidence refs, not free-prose fog. Templates are the system's answer to the question “how do we make useful outputs without letting LLM style overwrite custody?”

The current template organ is intentionally tiny: variable interpolation, simple loops, no expression engine, no network calls, no model calls, and render receipts. That is correct because artifact generation is a trust boundary. A good artifact is beautiful and useful, but its skeleton must be inspectable.

## 2. Sources

### Local sources

- `scripts/template_contract.py` — deterministic Jinja-like renderer with variables, simple for-loops, output path, context SHA, output SHA, receipt, and `model_calls_performed: False`.
- `scripts/case_packet_compiler.py` — creates case packets from custody, parse, timeline, staging, graph candidate refs, and receipt refs.
- `scripts/export_bundle.py` — scans for secrets before export, writes tar.gz bundle plus manifest, and verifies file hashes.
- `05_OUTPUTS/templates/` — scoped receipt/output zone for template renders.
- `core/language_membrane.py` — output weaving includes a deterministic template lane alongside RAG quotes, model synthesis, and smoothing.

### External Source Anchors

- Jinja's official documentation describes templates as placeholders/control structures rendered with passed data; LUCIDOTA intentionally implements only the subset it needs: <https://jinja.palletsprojects.com/>.
- W3C PROV-O supports recording which activity generated which artifact from which source entities: <https://www.w3.org/TR/prov-o/>.
- RFC 2119 anchors normative MUST/SHOULD language used in these RFCs and template contracts: <https://datatracker.ietf.org/doc/rfc2119/>.
- SEPIO's evidence/provenance/claim model supports keeping claim refs and evidence refs separate inside case packets: <https://ohsu.elsevierpure.com/en/publications/sepio-a-semantic-model-for-the-integration-and-analysis-of-scient/>.

## 3. Artifact Contract

An artifact generator SHOULD declare:

- artifact type,
- context input path or JSON,
- evidence refs,
- model involvement state,
- output path,
- context hash,
- output hash,
- receipt path,
- export/secret-scan state when leaving the local system.

A generated artifact is not correct because it “sounds right.” It is correct when the source refs, render context, generated bytes, and export checks can be inspected.

## 4. Whole-System Interaction

- **KORPUS/ETL:** supplies source and component refs.
- **Main spine:** requires artifact outputs to remain drafts/candidates unless a promotion/action gate authorizes them.
- **Diogenes:** external effects such as sending/publishing/exporting require command authority.
- **Language membrane:** combines deterministic templates with exact quotes and model synthesis, marking outbound state draft-only.
- **Local LLM fabric:** may provide draft text for a section, but the template contract controls artifact structure.
- **Indy_READs:** can produce briefs/reports/margin-note packets through product factory patterns.
- **Constant learning:** learning outputs can feed templates as verdicts/features, not hidden prose.
- **Filesystem law:** generated artifacts belong in scoped outputs, not mixed into source or doctrine unless promoted.

## 5. Benefit to the Whole System

Templates turn LUCIDOTA from a pile of candidates into a product factory. They let the operator live-code and iterate outputs while preserving evidence anchors. They also reduce LLM slop because the model can fill bounded lanes rather than inventing structure, citations, or authority.

The benefit is practical: case packets, export bundles, and review memos become reproducible. If a recipient challenges a claim, the operator can trace it to custody refs and receipt refs rather than defending a paragraph as vibes.

## 6. Correctness Argument

I believe this RFC is correct because the current code implements the desired boundary. `template_contract.py` performs deterministic substitution/loops and records hashes; it explicitly reports no model calls. `case_packet_compiler.py` builds artifacts from custody/parse/timeline/staging/graph refs, not free-floating summaries. `export_bundle.py` blocks on secret scan failure and verifies hash integrity.

The correctness property is reproducibility and traceability. If the same context and template render differently without explanation, the contract fails. If an output cannot identify its source refs, the artifact is not LUCIDOTA-grade no matter how polished it sounds.

## 7. Falsifiers

This RFC is wrong if:

- artifact generation depends on hidden model calls,
- outputs lack context/output hashes,
- case packets cannot trace claims to custody/parse/staging/graph refs,
- exports can bypass secret scans,
- templates become general code execution engines,
- generated prose is allowed to upgrade candidate claims into facts.

## 8. Filesystem / Runtime Consequences

- Keep templates and generated artifacts in scoped template/output directories.
- Store receipts for renders, packets, and exports.
- Keep source refs inside artifacts wherever possible.
- External publication/export/sending requires explicit command authority.
- Promote only stable template contracts into canonical docs; experimental templates remain dev-library/artifact-lab material. A template is allowed to be plain and small if it preserves source refs, because repeatable plain output beats beautiful untraceable prose.
