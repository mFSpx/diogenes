## 1. Thesis

LUCIDOTA should be understood as a local sovereign intelligence operating system for hypersystemic work. It is not valuable because it has many scripts. It is valuable if those scripts behave like composable organs: predictable in execution, explicit in authority, bounded by receipts, able to ingest chaos, able to reason abductively, and unwilling to promote unlabeled bullshit into truth.

My current belief is that this thesis is true because the repository already contains repeated independent evidence of the same architecture:

- a doctrine layer saying blueprint first and receipts over claims,
- a queue spine built around ABSURD/Postgres,
- a custody/ETL/KORPUS layer,
- an ontology/graph layer,
- a model-runtime layer constrained by local hardware,
- a dev-library layer for reuse,
- named organs such as Diogenes, Indy_READs, PercyphonAI, and Krampus Express.

The failure mode is not absence of vision. The failure mode is scattering, name drift, unpromoted source material, and artifacts sitting in the repo with the wrong authority gravity.

## 2. RFC Standard

This program uses RFC-style normative language. `MUST`, `SHOULD`, and `MAY` follow the RFC 2119 convention as a source model, but local LUCIDOTA receipts and gates are stronger than prose.

Every subject RFC MUST include:

- thesis,
- sources,
- local evidence,
- whole-system interaction,
- whole-system benefit,
- correctness argument,
- falsifiers,
- filesystem/runtime consequences.

## 5. Correctness Standard

A subject is not done when it sounds right. It is done when:

- the relevant source files have been inspected,
- contradictions are named,
- claims are backed by source paths or URLs,
- the RFC states what would falsify it,
- the filesystem/runtime consequence has been applied or explicitly queued,
- a receipt records the work.
